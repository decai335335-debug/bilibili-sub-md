# DEV_LOG —— bilibili-sub-md

开发日志：记录项目起源、迭代决策、踩坑与方案选择。

---

## 1. 项目起源

**原始需求来源：** 用户在日常学习中大量使用 Bilibili 学习技术课程和讲座，希望像整理 YouTube 字幕一样，把 B 站视频字幕批量下载到 Obsidian 笔记库中。同时已经有一个成熟的 Chrome 插件 [Bilibili Obsidian Clipper](https://github.com/haixiong1997/Bilibili-Obsidian-Clipper) 可以单个视频抓取字幕，但缺乏批量能力。

**核心痛点：**
1. B 站 UP 主经常发布 20~50 集的系列课程，手动逐集复制字幕太耗时
2. 收藏夹里存了大量学习视频，想批量提取字幕做知识库
3. 多 P 视频的字幕分散在不同 CID 下，需要按 P 拆分保存
4. 不同视频的字幕类型不同：有人工字幕、AI 字幕、多语言字幕，需要智能选择

**项目目标：** 基于 Bilibili Obsidian Clipper 的字幕抓取逻辑，做一个命令行批量下载工具，支持视频链接、多 P 视频、合集、收藏夹的统一处理。

## 3. 踩坑记录

| 问题 | 根因 | 解决方案 | 涉及版本 |
|------|------|---------|---------|
| B 站 API 返回 `code: -509` 请求过于频繁 | 短时间内多次调用 Bilibili API 触发限流 | 增加重试机制、随机延迟 1~2.5 秒、默认 3 并发 | v0.1.0 |
| 字幕 URL 以 `//` 开头 | B 站 API 返回的 subtitle_url 是协议相对路径 | 统一规范化：`//` → `https:`，无协议 → `https://` | v0.1.0 |
| 多 P 视频 CID 选择错误 | `?p=3` 需要对应第 3 个 page 的 CID，而不是默认 CID | 通过 `extract_page_index` 从 URL 解析 p 参数，再用 `pickPageFromPages` 匹配 | v0.1.0 |
| AI 字幕 lan 字段以 `ai-` 开头 | B 站的 AI 自动生成字幕用 `ai-zh` 等标识 | 在 `subtitlePriority` 中降低 AI 字幕优先级，除非没有人工字幕 | v0.1.0 |
| 合集 API 分页 | B 站系列/合集接口一次最多返回 30 条 | 循环请求 `pn` 直到返回空数组 | v0.1.0 |

## 4. 设计决策

### 决策一：为什么复刻 Bilibili Obsidian Clipper 的 API 逻辑？

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 从头研究 B 站 API | 完全自主可控 | 时间成本高，接口变化快 | ❌ 不选 |
| 使用 yt-dlp 解析 B 站 | 成熟工具，一站式 | B 站字幕解析不稳定，且不支持合集批量 | ❌ 不选 |
| **复刻 BOC 插件的 API 调用链** | API 已被验证稳定；有完整的字幕选择、时长校验逻辑；容易扩展批量功能 | 依赖 B 站公开 API，未来可能变化 | ✅ **选中** |

### 决策二：为什么播放列表识别采用"URL 模式匹配"而不是调用统一接口？

B 站的"播放列表"是一个泛化概念，实际包含多种类型：
- UP 主合集：`channel/collectiondetail?sid=...`
- 频道系列：`channel/seriesdetail?sid=...`
- 收藏夹：`/list/ml{mlid}`
- 稍后再看：`/list/watchlater`

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 调用统一接口 | 代码简洁 | B 站没有统一的"播放列表"接口，不同类型的分页参数和返回结构完全不同 | ❌ 不选 |
| **URL 模式匹配 + 分类处理** | 灵活适配各种播放列表类型；用户只需粘贴链接，零心智负担 | 新增类型需要新增正则和 API 处理 | ✅ **选中** |

### 决策三：为什么字幕选择优先级是"中文 → 英文 → AI 字幕"？

**背景**：B 站视频的字幕通常有：UP 主上传的人工字幕、B 站 AI 自动生成的字幕、多语言字幕。

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 始终选第一个字幕轨 | 逻辑简单 | 第一个是 AI 字幕时质量差；第一个是非目标语言时体验差 | ❌ 不选 |
| 让用户每次指定 | 最精确 | 批量下载时体验极差 | ❌ 不选 |
| **中文优先 → 英文优先 → AI 兜底** | 符合中文用户主要使用场景；AI 字幕作为无人工字幕时的 fallback | 非中文/英文用户需要手动指定 `--lang` | ✅ **选中** |

### 决策四：为什么默认输出 Markdown 而不是 SRT？

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 默认 SRT | 兼容剪辑软件 | 用户主要场景是阅读、整理、搜索，SRT 时间轴是噪音 | ❌ 不选 |
| **默认 Markdown** | 直接可用作 Obsidian 笔记；保留时间戳但不过度干扰阅读；支持 YAML frontmatter | 剪辑软件需要再转 SRT | ✅ **选中** |
